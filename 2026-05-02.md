## Local LLM 환경 지식 네트워크(Knowledge Graph) 반영 및 추론 활용 기술 설계안

본 문서는 로컬 LLM 환경에서 지식 네트워크(KG)를 효율적으로 반영하고 이를 추론에 활용하기 위한 구체적인 툴 사용법, 프레임워크 및 구현 방법론을 조사하여 기술적 설계안을 제시합니다.

### 1. 핵심 목표 요약 (Goal Summary)

*   **목표:** 로컬 LLM이 외부 지식 네트워크를 참조하여 보다 정확하고 추론 기반의 답변을 생성하도록 시스템을 구축한다.
*   **핵심 방법론:** 데이터 임베딩 및 그래프 구조화 $\rightarrow$ RAG(Retrieval-Augmented Generation) 프레임워크 적용 $\rightarrow$ 그래프 기반 추론 모듈 통합.

### 2. 추천 프레임워크 및 기술 스택 (Frameworks & Stack)

| 영역 | 추천 기술/프레임워크 | 선택 이유 |
| :--- | :--- | :--- |
| **지식 인덱싱/벡터화** | LlamaIndex, LangChain | LLM과 외부 데이터 소스 간의 연결(Indexing, Retrieval)을 표준화하여 구현 용이성 확보. |
| **지식 그래프 구축 (KG)** | Neo4j (Local Instance), RDFLib | 관계형 데이터가 아닌 복잡한 관계 기반 지식을 구조화하고 탐색(Traversal)에 최적화. 로컬 환경 구동 가능성 고려. |
| **LLM 통합 및 추론** | LangChain (Graph Integration Module) | 벡터 DB/KG 데이터와 LLM을 연결하는 오케스트레이션 레이어 제공. 그래프 쿼리를 프롬프트에 효과적으로 주입하는 모듈 설계 용이. |
| **로컬 환경 최적화** | Ollama, Llama.cpp 기반 모델 | 로컬 LLM 구동 환경과의 호환성을 극대화하여 인프라 부담 최소화. |

### 3. 지식 네트워크 반영 및 구현 방법론 (Implementation Methodology)

#### 단계 1: 지식 데이터의 구조화 및 KG 구축 (Data Structuring & KG Construction)

1.  **데이터 수집 및 정제:** 비정형 문서(PDF, TXT 등)에서 핵심 사실(Entity), 관계(Relation), 속성(Attribute)을 추출합니다.
2.  **엔티티-관계 정의:** 도메인 특화 용어(Ontology)를 정의하여 지식의 스키마를 설정합니다. (예: `[개체] A`는 `[관계] B`와 연결됨).
3.  **KG 변환:** 추출된 데이터를 Neo4j 또는 RDF 형태로 변환하여 그래프 데이터베이스에 저장합니다. **(핵심)**
    *   *구현 방법:* LlamaIndex의 지식 그래프 모듈을 활용하거나, Python의 `networkx` 라이브러리를 사용하여 초기 구조를 설계한 후 DB로 마이그레이션합니다.

#### 단계 2: LLM과의 연결 및 임베딩 (LLM Integration & Embedding)

1.  **텍스트 임베딩:** 원본 문서와 KG 노드/엣지 정보를 벡터 임베딩 모델(예: Sentence Transformers, 혹은 로컬에서 구동 가능한 Embedding Model)을 사용하여 벡터 공간에 매핑합니다.
2.  **하이브리드 인덱스 구축:** 검색 효율성을 위해 **벡터 검색 (RAG)**과 **그래프 검색 (Graph Traversal)**을 병행하는 하이브리드 인덱스를 구축합니다.
3.  **KG-RAG 설계:** 사용자의 질문이 들어왔을 때, LLM은 단순히 벡터 검색 결과뿐만 아니라 KG에서 정의된 관계를 따라 추론하여 문맥을 확장하도록 프롬프트에 명시적으로 지시하는 시스템을 설계합니다.

#### 단계 3: 추론 및 반영 (Inference & Reflection)

1.  **그래프 기반 질의:** 사용자 질문이 들어오면, LLM은 먼저 질문을 KG 내의 관계 패턴으로 변환하여 가장 관련성 높은 노드 경로를 검색합니다.
2.  **추론 생성:** 검색된 그래프 경로(Context)와 벡터 임베딩 결과를 종합하여 최종 답변을 생성합니다. (베이지안 추론의 원칙 적용: 불확실성을 반영한 최적 경로 탐색).

### 4. 기술 설계 요약 (Technical Design Summary)

| 구성 요소 | 역할 | 입력 데이터 | 출력 결과물 |
| :--- | :--- | :--- | :--- |
| **Knowledge Ingestion Layer** | 원본 데이터를 KG 구조로 변환 | 비정형 문서, 도메인 용어집 | Neo4j/RDF 그래프 DB |
| **Retrieval Layer (RAG)** | 관련 지식 검색 | 사용자 질의, 벡터 임베딩 | 가장 관련성 높은 노드 집합 및 텍스트 청크 |
| **Inference Engine** | 추론 실행 및 결과 통합 | RAG 결과 + KG 경로 정보 | 통계적 근거 기반의 최종 답변 |

## 2. 아키텍처 다이어그램 및 데이터 플로우 (The Blueprint)

```mermaid
graph TD
    A[사용자 브라우저/Next.js Frontend] -->|1. 사용자 상호작용: 키워드 A & B 입력| B(Next.js API Route: /api/query-knowledge);
    B -->|2. 데이터 요청 및 검증| C{Backend Logic Layer (Node.js)};
    C -->|3. Cypher 쿼리 실행| D[Neo4j Database];
    D -->|4. 그래프 결과 반환: Node List, Edge List, Relationship Data| C;
    C -->|5. 데이터 정제 및 포맷팅 (JSON/Graph Schema)| B;
    B -->|6. 최종 시각화 데이터 전송| A;
    A --> F[React Flow Component: KG Vis Module];
```

**핵심 로직 흐름 설명:**
1. **사용자 입력:** 사용자가 웹사이트의 '사고의 지도' 인터랙티브 섹션에서 키워드(Source Node, Target Node)를 지정하고 '탐색' 버튼을 누릅니다.
2. **API 호출:** Next.js 프론트엔드는 이 요청을 `fetch('/api/query-knowledge', { body: { source: 'A', target: 'B' } })` 형태로 백엔드 API 라우트로 보냅니다.
3. **데이터 처리 (Node.js):** API Route는 들어온 키워드를 검증하고, Neo4j 드라이버를 사용하여 Cypher 쿼리를 생성합니다.
4. **Neo4j 쿼리:** DB가 관계성(Relationship)을 탐색하여 관련 노드와 엣지 데이터를 반환합니다.
5. **데이터 포맷팅:** API Route는 받은 원시 그래프 데이터를 `[ {id: string, label: string, x: number, y: number}, ... ]` 형태의 React Flow 친화적인 JSON 배열로 변환합니다.
6. **렌더링:** 프론트엔드는 이 깔끔한 데이터를 받아 React Flow 컴포넌트에 전달하고, 인터랙티브하게 그래프를 그려 사용자에게 보여줍니다.

### B. `query-knowledge.ts` (백엔드 로직 - **실행 가능한 코드 예시**)

이 파일은 Next.js API Route(`/api/query-knowledge`) 내부에 위치하며, 데이터베이스와 통신하는 핵심입니다.

```typescript
// 💡 가정: neo4jClient는 Neo4j 드라이버를 통해 초기화된 클라이언트 객체입니다.

export default async function handler(req: NextApiRequest, res: NextApiResponse) {
  const { source, target } = req.body;

  if (!source || !target) {
    return res.status(400).json({ error: "Source and Target keywords are required." });
  }

  try {
    // 1. Cypher 쿼리 작성 (관계 탐색 및 제약 조건 포함)
    const cypherQuery = `
      MATCH (a:Concept {name: $source})-[r*2..4]->(b:Concept {name: $target})
      RETURN a, r, b
    `;

    // 2. Neo4j 실행 및 결과 수신
    const session = neo4jClient.session();
    const result = await session.run(cypherQuery, { source: source, target: target });
    const records = result.records;

    // 3. 데이터 포맷팅 (React Flow 친화적 JSON 구조로 변환)
    const nodes: any[] = []; // 노드 리스트 저장용 배열
    const edges: any[] = []; // 간선 리스트 저장용 배열
    
    for (const record of records) {
        // [구현 상세]: 각 레코드에서 노드와 관계를 추출하여 React Flow의 Node/Edge 형식에 맞게 변환합니다.
        // 예시: 2단계 추론 과정에 참여한 모든 개념을 노드로, 그 연결을 간선으로 만듭니다.
        const sourceNode = record.get('a').properties; // A
        const targetNode = record.get('b').properties; // B

        nodes.push({ id: `${source}-${target}`, data: { label: '추론된 경로', type: 'path' }, position: { x: Math.random() * 500, y: Math.random() } });
        edges.push({ id: `edge-${Math.random()}`, source: sourceNode.name, target: targetNode.name, data: { label: record.get('r').type || '연결' } });
    }

    // 4. 최종 응답
    return res.status(200).json({ nodes, edges, count: records.length });

  } catch (error) {
    console.error("Neo4j Query Error:", error);
    return res.status(500).json({ error: "Graph query failed or DB connection lost." });
  }
}
```

### ⚙️ 2. 컴포넌트별 상세 명세서 및 상호작용 로직

#### 🔵 Component A: `<QueryInput />` (사용자 질문 입력창)

*   **목적:** 사용자의 개방형 질문을 받아 추론 과정의 트리거(Trigger)로 사용합니다.
*   **HTML/React 구조:** `div > input[type="text"] + button` 형태로 구성.
*   **디자인 토큰:**
    *   `Border-Radius`: 8px (부드러운 학술적 느낌)
    *   `Font-Family`: Noto Serif KR, 16px
    *   `Color-Background`: #F9FAFB (미세한 배경색으로 집중 유도)
*   **상태 및 액션:**
    1.  **기본 상태:** 플레이스홀더 텍스트: "예시 질문을 입력해 보세요. (불확실성을 낮추는 방법?)"
    2.  **Focus 상태:** 테두리(Border)가 `Color-Primary`(#0A5E9B, 신뢰감 있는 네이비 계열)로 하이라이트됩니다.
    3.  **클릭/제출 (`onClick`/`onSubmit`):** 버튼 클릭 시 로딩 애니메이션이 활성화되며, 컴포넌트를 `<PathRenderer />`에게 쿼리 데이터를 전달합니다.

#### 🟣 Component B: `<PathRenderer />` (추론 경로 시각화 애니메이션)

*   **목적:** 질문을 던진 것부터 최종 결론에 도달하는 '사고 과정' 자체를 눈으로 확인시켜 주는 것이 핵심입니다.
*   **상태 및 흐름 (State Machine):**
    1.  **`IDLE`:** 기본 상태. 비활성화되어 있음.
    2.  **`QUERYING` (애니메이션 1 - 입력):** 사용자가 질문을 제출하면, 먼저 **질문 키워드와 가장 연관성이 높은 핵심 노드(Node A)**에서 발광하는 듯한 애니메이션이 발생합니다. (Source Node 강조)
    3.  **`TRACING` (애니메이션 2 - 추론 경로):** 선택된 노드들 사이의 연결 관계(`Edge`)를 따라 빛나는 **방향성 화살표(Directed Arrow)**가 순차적으로 그려집니다. 이 과정은 1~2초에 걸쳐 진행되어야 합니다.
    4.  **`RESULT` (애니메이션 3 - 결과 확정):** 마지막 목적지 노드(최종 결론) 전체가 잠시 동안 **펄스(Pulse)** 애니메이션을 일으키며, 주변의 관련 개념 노드가 점차 활성화되면서 그래프에 배치됩니다.
*   **기술 요구사항:** React Transition Group 또는 SVG animation 라이브러리를 활용하여 부드럽고 명확한 시퀀스를 구현해야 합니다.

#### 🟢 Component C: `<KnowledgeGraphVisualizer />` (노드 깊이 및 색상 규칙)

노드의 크기(Size)와 색상(Color)은 해당 노드가 지식 체계 내에서 어느 정도의 **근거 수준(Depth)**을 갖는지를 나타내야 합니다.

| Depth 레벨 | 의미 | 시각적 표현 (CSS/Token) | Size Rule |
| :--- | :--- | :--- | :--- |
| **Level 1** | **핵심 개념 (Core Concept)** | `Color: #0A5E9B` (Primary). 가장 높은 신뢰도. | Large (지름: 30px) |
| **Level 2** | **하위 개념/원리 (Sub-Concept/Principle)** | `Color: #4F8BA7` (Secondary). 주요 지식의 분기점. | Medium (지름: 20px) |
| **Level 3+** | **세부 근거/예시 (Detail/Evidence)** | `Color: #9BBCC6` (Tertiary). 구체적인 뒷받침 자료. | Small (지름: 15px) |
| **Inactive** | 미사용 노드 | `Opacity: 0.4`, `Cursor: pointer` | Size Rule 유지 |

*   **상호작용:** 모든 노드는 클릭 가능해야 하며, 클릭 시 해당 노드를 중심으로 연결된 관계(`Edge`)들이 잠시 강조되어야 합니다. (Focus Effect)
